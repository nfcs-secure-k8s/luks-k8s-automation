# Vault Integration

### Vault Integration for Key Management and Rotation

In order for different institutions to manage their own encryption keys, we move beyond static or local storage methods and integrate with HashiCorp Vault. This enables centralized key storage, version tracking, controlled ownership of keys, and auditable key lifecycle management.

### Purpose of Vault Integration

The initial proof of concept we introduced earlier demonstrated that block storage can be encrypted using Kubernetes, an operator, and LUKS. However, this does not provide the level of governance, auditability, and controlled life cycle required in a Trusted Research Environment.
The Vault integration was therefore introduced to address three requirements:

- centralized key storage
- controlled key rotation
- secure deletion of keys when volumes are destroyed

### Implementation Design

Every encrypted volume uses the KV v2 secrets engine in HashiCorp Vault, following a tenancy-aware path structure: `/secret/tenants/<institution>/luks-key/<volume-name>`.

When a new volume is created, the Operator checks if a key exists for that volume and if not it creates one using a new 256-bits value using `secrets.token_hex(32)` and stores it in vault. This creates a unique per volume key while allowing key ownership to be separated from storage backends.

A design decision was made for initial key per volume to be generated by the Operator but **the user is required to replace** the initial key with their own value in the Vault UI. This is to ensure that each institution manage the ownership of their keys rather than being managed by the platform owner.

### Vault Version Synchronisation

To detect vault key changes by the user or system, the Operator uses a synchronisation function to detect the vault key version change. It uses the Kopf timer function for detection, which runs every 30 seconds and checks for a vault version change.
The Operator reads the current secret metadata from the vault to retrieve the KV version number associated with the stored key. The version number is then written back to the custom resource in two places:

- as a metadata `annotation: vaultversion`
- as a status field: `current_vault_version`

This mechanism provides the following benefits:

- Enables the Operator to have visibility on the vault-backend key state
- Provides the Operator a Kubernetes-native trigger point for vault key changes and rotation.

### Operator-Driven Key Rotation

The second controller function watches for changes in the metadata.annotations.vaultversion field for a version number change. When the Operator detects a version number increase, it knows a key change has been made and then dispatches a job to perform the following:

1. reads the new key from the latest Vault version
2. reads the previous key from the prior Vault version
3. identifies the node currently associated with the encrypted volume
4. dispatches a Kubernetes Job to that node
5. performs LUKS rekeying using:

- `cryptsetup luksAddKey`
- followed by `cryptsetup luksRemoveKey`

After the key rotation is complete, the encrypted volume is updated in place. The Operator records the processed version in the resource status so that the rotation is not repeated unnecessarily.
